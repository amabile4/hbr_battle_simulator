## 1. 出発点（第1稿と共通）

以下の問題意識は第1稿と同じ:

1. 再計算時に自動判定が復活し、手動意図と衝突し得る
2. save/load 往復後も同じ体験を維持したい
3. 自動計算を活かしつつ、過度な自動化で行動入力と結果が乖離するのは避けたい

## 2. 第2稿の基本思想

第2稿では、以下を原則とする:

1. リプレイJSONは「操作したことだけ」を記録する
2. 計算結果（累積DP/HP/破壊率、自動ブレイク/討伐確定結果）は記録しない
3. 手入力でのDP/HP/破壊率/パラメータ調整は、その場のプレビュー専用で永続化しない
4. 自動判定は「ガイド表示」を基本とし、確定はユーザー操作で行う

## 3. 懸念シナリオ（方針分岐の根拠）

例: 2連戦で1体目撃破後に2体目が召喚されるケース

- 元条件では5T撃破、6Tから次敵出現
- パラメータ変更後の再計算で3T撃破になったとしても、4T出現へ自動前倒しすると行動入力との整合が崩れる

第2稿の判断:

- 出現タイミング・確定的な戦況遷移は、操作イベントに従って保持する
- 自動計算は補助情報として見せるが、操作履歴を書き換える側に使わない

## 4. モード再定義（第2稿）

優先度順:

1. 手動計算（プレビュー専用）
2. 自動計算ハイブリッド（ガイド表示）
3. 自動計算（参照ビュー）

### 4.1 手動計算（最優先）

- ユーザーが現DP/現HP/現破壊率/現パラメータ等を一時入力して、そのターンの見え方を確認
- 入力は「一時表示」であり、commit対象にもJSON保存対象にも含めない

### 4.2 自動計算ハイブリッド

- 累積計算により「このタイミングでDP割れそう/討伐しそう」をガイド表示
- ただし確定状態は操作で指定されるまでfixしない
- 自動結果は提案、手動結果が確定

### 4.3 自動計算

- 純粋な自動積算結果を参照するビュー
- 手動確定表示を一時的に非表示にして比較表示する用途は許容
- ただし表示切替は保存しない（ビュー状態）

## 5. 永続化ポリシー（第2稿）

保存するもの:

1. ユーザー操作イベント（既存 replay operation / override）
2. その操作に必要な入力値

保存しないもの:

1. 計算途中の累積結果
2. 自動判定の抑止情報（suppression）
3. プレビュー入力値（手動計算モードの一時調整）

補足:

- suppression永続化は第1稿の有力案だが、第2稿では「理念との整合」と「複雑化回避」を優先して採用しない

## 6. 仕様差分（第1稿との比較）

1. 第1稿: suppressionを保存して再計算後も自動復活を抑える
2. 第2稿: suppressionを保存せず、操作イベントのみ保存する
3. 第1稿: mode + suppression中心のデータモデル拡張
4. 第2稿: modeはビュー/計算補助として扱い、保存は最小化

## WBS-B: データモデル整理（縮小方針）

1. B-1 既存 replay operation / override を正本として再確認
2. B-2 suppression永続化案を採用しない設計判断を記録
3. B-3 save/loadで「操作のみ保存」を回帰テストで固定

完了条件:

- 保存JSONに計算結果・抑止情報が混入しない

## WBS-C: 計算表示レイヤ

1. C-1 自動積算結果を「ガイド表示」として分離
2. C-2 手動確定状態と自動ガイドを並置表示（競合時は手動を優先表示）
3. C-3 手動計算プレビュー入力をターン内一時状態として実装（非保存）

完了条件:

- 再計算しても操作履歴を侵食せず、比較表示が可能

## WBS-D: 連戦・召喚整合

1. D-1 連戦/召喚シナリオで「操作起点の遷移」を保持
2. D-2 再計算で撃破ターンが前倒しされても、召喚イベントが勝手に前倒しされないことを確認
3. D-3 必要なら「自動結果との差分警告」を表示

完了条件:

- 行動入力と戦況遷移の整合が保たれる

## WBS-E: テスト

1. E-1 unit: 非永続プレビュー状態の寿命管理
2. E-2 integration: save→load→recalculateで操作履歴のみ維持
3. E-3 integration: 連戦召喚シナリオで遷移前倒しが起きない
4. E-4 e2e: 手動確定と自動ガイドの切替・比較表示

完了条件:

- 代表シナリオが安定再現し、JSON理念との矛盾がない

## 10. 最新要望整理（2026-06-11追記）

### 10.1 やりたいこと

1. DPダメージ、HPダメージ、破壊率計算の自動計算
2. DPダメージ、HPダメージ、破壊率計算の累積化
3. 自動ブレイク、自動討伐の組み込み

### 10.2 懸念

- 想定外のタイミングでブレイク/討伐が確定し、操作意図と結果がズレること

### 10.3 守りたい理念

1. このシミュレーターは「プレイヤーが操作したことを記録する」ことを正本とする
2. JSON保存/読込後、設定変更して再計算することで流れを俯瞰できることを優先する
3. 多少の計算上の粗さ（例: SPマイナス許容）よりも、運用の単純さと互換性を優先する

### 10.4 操作モデル（第2稿寄り）

1. 手動操作状態
- ブレイク/討伐を手動入力したら、その入力を正とする
- 自動計算が「ここでDP0になる」と示していても、手動ブレイク入力がなければ確定しない
- 逆にDP残量があっても手動ブレイク入力があれば、そこでブレイク確定として扱う

2. 自動操作状態
- 累積計算でDP0/HP0到達時に自動でブレイク/討伐を確定する参照モード
- Eシールドは既存どおり自動運用を基本とする

3. 一時比較ビュー
- 手動ブレイク/討伐指定を一括で「一時的に非表示・無効化」して、自動計算ならどこで割れるかを確認できるようにする
- これは表示上の切替であり、保存対象にはしない

## 11. 第1稿/第2稿 比較メモ（現時点）

1. 共通
- 自動計算の有用性は認める
- save/load/recalculate 後の一貫性を重視する

2. 分岐点
- 第1稿: 自動復活抑止のため suppression などの状態を保存寄りで拡張
- 第2稿: 保存は操作イベントのみ。自動結果はガイド/比較用途で非保存

3. 現在の暫定判断
- 本要望は第2稿寄り（操作正本 + ガイド表示 + 一時比較ビュー）
- 第1稿は対案として保持し、最終決定前に比較検証する

4. 検証観点
- 連戦召喚シナリオでの整合
- 手動入力と自動ガイドの説明可能性
- JSON互換性と保守コスト
