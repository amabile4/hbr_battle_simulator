# 新規スタイル監査運用

**ステータス**: 🟢 進行中  
**最終更新**: 2026-07-02

## 目的

新しい `json/` / `assets/` データを受領した後、最新 `in_date` のスタイルがシミュレーター上で不足なく機能するかを、スタイル単位で監査する。

この監査は、外部で解析されたデータを受け取った後の受入確認であり、データ取得元や解析元の固有名はリポジトリ内に残さない。

## 監査手順

1. データ入口を確認する
   - `styles.json` の `in_date` 空欄件数を確認する。
   - 空欄があれば完全新規スタイルとして扱い、抽出側の `in_date` 付与漏れとして注意する。
   - 最大 `in_date` を特定し、その日時に追加されたスタイル一覧を監査キューにする。

2. スタイル単位で棚卸しする
   - スタイル説明、ロール、属性、スキル、追撃、パッシブ、限界突破パッシブを列挙する。
   - `cond` / `overwrite_cond` / `iuc_cond` / `target_condition` / `hit_condition` を抽出する。
   - `SpecialStatusCountByType(...)`、状態変化バフ、状態異常デバフ、追加ターン、スキル回数、スキル内パッシブを確認対象に含める。

3. 未知要素を分類する
   - 条件式は AST evaluator で解決可能か確認する。
   - `skill_type` は既存データでの出現履歴と実装接続を確認する。
   - 未知または未接続の可能性があるものは、`OK` / `実装ギャップ` / `データ不備` / `要実測確認` に分類する。

4. 挙動を検証する
   - SP 境界条件、対象条件、敵弱点条件、追加ターン条件、永続デバフ、限界突破パッシブを個別に確認する。
   - 既存テストがある場合は対象テストを実行する。
   - テストが不足している場合は、監査用 probe で再現し、必要なら後続の実装タスクで regression test を追加する。

5. スタイル説明と照合する
   - 説明文で約束されている機能が、シミュレーター上で再現できるか確認する。
   - 説明文に直接出ない限界突破パッシブも、スタイルの完成度として別途確認する。

### 実装ギャップ

`GiveDefenseDebuffUp` は passive event として記録されるが、敵デバフ効果量の解決に反映されていない。

監査 probe では、`華炎` の `DefenseDown` power が LB1 / LB2 / LB3 いずれも `0.2` のままだった。

| LB | 発火イベント | 実際の DefenseDown power | 期待される扱い |
|----|--------------|--------------------------|----------------|
| LB1 | なし | `0.2` | `0.2` |
| LB2 | `侵食(小)` +15% | `0.2` | 防御力ダウン効果量へ反映 |
| LB3 | `侵食(小)` +15%、`侵食[アビリティ強化]` +25% | `0.2` | `侵食[アビリティ強化]` +25% のみを反映（下記仕様） |

仕様（確定）:

- 同じ限界突破段階で付与される同種の AUTO パッシブスキルの効果値が複数存在し重複する場合、加算ではなく、その中で最も高い効果値のみが採用される。
- 本ケースでは `侵食(小)` +15% と `侵食[アビリティ強化]` +25% が同種（防御力ダウン効果量アップ）のため、加算した `+40%` ではなく、最も高い `侵食[アビリティ強化]` +25% のみが採用される。
- ただし現状はいずれの値も効果量計算へ反映されていないため、実装ギャップは解消していない。

### 判定

スタイル説明にある SP 条件付きデバフ運用は概ね再現できている。

一方で、限界突破パッシブ `侵食` が実際の防御力ダウン効果量へ反映されないため、「すべてのスタイルが十全にシミュレーター上で機能するか」という観点では未達。後続タスクとして `GiveDefenseDebuffUp` を敵デバフ効果量 resolver に接続し、LB2/LB3 の regression test を追加する必要がある。
