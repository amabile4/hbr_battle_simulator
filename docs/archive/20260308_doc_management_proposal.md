# ドキュメント管理ルール 提案書

**作成日**: 2026-03-08
**ステータス**: ✅ 合意済み（2026-03-08）
**後継先**: 合意内容を `docs/README.md` に反映し、`AGENTS.md` / `CLAUDE.md` には参照ポインタのみ追記

## 背景・問題

`docs/` 以下が以下の種類のファイルが混在し、どれが「今有効な情報か」が判断できない状態になっている。

| 実態 | 例 |
|------|----|
| 進行中のタスク | `passive_implementation_tasklist.md` |
| 確定設計（参照専用） | `specs/runs/RUN_20260228_001/` |
| 調査スナップショット | `help-research/`, `20260306_tasklist/` |
| 廃止済み設計 | `rebuild-spec/`, `rebuild-spec-v2/` |

Codex が `docs/` の実装計画ファイルを参照しながら開発を進めているため、
ドキュメントが実態と乖離した場合にそのまま古い情報で実装が進むリスクがある。

---

## 提案するルール

### ルール1: `docs/README.md` を唯一のナビゲーション起点とする

`docs/README.md` を常に最新状態に維持し、全ドキュメントのステータスを一覧できる **マスターインデックス** とする。

- **実装者（Codex / Claude）は、実装完了時に該当ドキュメントのステータスを必ず更新する**
- `docs/README.md` に載っていないドキュメントは参照不要（実質アーカイブ）と見なせる
- AIエージェントが `docs/` を参照する際は、まず `docs/README.md` から始める

### ルール2: ドキュメントのステータスを先頭行に明記する

各ドキュメントの冒頭に、以下の形式で1行ステータスを記載する。

```markdown
> **ステータス**: 🟢 進行中 | 📅 最終更新: 2026-03-08
```

#### ステータス定義

| 記号 | ラベル | 意味 | 更新タイミング |
|------|--------|------|----------------|
| 🟢 | 進行中 | アクティブに更新するタスク・計画 | 実装進捗に合わせて随時 |
| 📚 | 参照 | 変わらない確定仕様（読む専用） | 仕様変更時のみ |
| 📦 | スナップショット | 特定時点の調査記録（変更しない） | 変更しない |
| ✅ | 完了 | タスク完了・実装済み | 完了確認時 |
| 🗄️ | アーカイブ | 廃止・後継ドキュメントあり | アーカイブ移動時 |

### ルール3: ドキュメントの種類ごとに配置ルールを定める

```
docs/
├── README.md                      # マスターインデックス（常に最新）
│
├── [テーマ別フォルダ]/             # 生きているドキュメント群
│   ├── code-review/               # コードレビュー結果
│   ├── specs/                     # 確定設計仕様
│   └── help-research/             # ゲーム仕様調査（スナップショット）
│
├── [YYYYMMDD]_[テーマ]/           # 日付プレフィックス = スナップショット
│   └── 20260306_tasklist/         # 変更しない・参照のみ
│
└── archive/                       # 廃止済み（移動のみ、削除しない）
    ├── rebuild-spec/
    └── rebuild-spec-v2/
```

**配置ルール**:
- 調査・タスク一覧など「その時点限り」の成果物は `YYYYMMDD_` プレフィックスを付けてルート直下に置く
- 廃止・後継が出たドキュメントは `archive/` に移動する（削除しない）
- 進行中の計画書はテーマ別フォルダか `docs/` ルートに置く

### ルール4: 実装完了時にドキュメントを更新する（AIエージェント必須）

AIエージェント（Codex / Claude）が実装タスクを完了した際は、以下をセットで実施する：

1. **対象ドキュメントのステータスを更新する**
   - タスク完了 → `🟢 進行中` → `✅ 完了` に変更
   - 部分完了 → 完了した項目にチェック `[x]` を入れる

2. **`docs/README.md` のステータス列を更新する**
   - 該当行のステータス記号・最終更新日を変更する

3. **超過した実装計画は削除ではなく完了マークを残す**
   - 誰が見ても「実装済みかどうか」が分かるよう、完了項目を消さずに `[x]` で残す

## AGENTS.md / CLAUDE.md への追記方針（合意済み）

### 方針: thin pointer（参照ポインタのみ）

**詳細ルールは `docs/README.md` に集約し、AGENTS.md/CLAUDE.md には1〜2行のポインタのみ追記する。**

この方針を採用する理由：
- AGENTS.md/CLAUDE.md はセッション開始時に自動でコンテキストに読み込まれるため、
  肥大化させると重要指示が埋もれる
- ルールを複数ファイルに分散させると更新漏れが発生する
- `docs/README.md` を single source of truth にすることで保守コストを下げる

#### AGENTS.md / CLAUDE.md に追記する内容

```markdown
## Documentation Conventions

- `docs/` 内のファイルを参照・更新する際は、必ず `docs/README.md` を確認し、
  記載のドキュメント管理ルールに従うこと。
- 実装タスクの完了時は、対応ドキュメントのステータス更新を実装とセットで行うこと（必須）。
```

これだけ。詳細は `docs/README.md` に委譲する。
