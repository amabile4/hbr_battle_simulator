## 📋 フェーズ別タスク・成果物・完了条件

### 📂 フェーズ1: 数式モデルの確立とデータ補完エンジンの構築 (Python)

シミュレータ側が持っていない「キャラクター能力値 (STR/DEX等)」や「敵の防御値」のギャップを埋めるための自動補完エンジンをPython側で開発します。

#### T1.1: Excelからのダメージ計算数式の完全抽出とPythonコード化
- **内容**: `HBR計算機🎭.xlsx` の `W59` (基礎通常)、`X59` (基礎クリティカル) 等の数式をPythonで計算可能なロジックとして再構成します。
- **インプット**: [damage_calculation_model.md](file:///Users/ram4/git/hbr_calc/docs/damage_calculation_model.md) の数式仕様。
- **アウトプット**: Python上の計算モジュール `damage_calc_engine.py`。
- **完了条件**: Excel内の代表的な数値入力に対して、Pythonモジュールが同一の基礎ダメージ値を出力できること。

#### T1.2: キャラクター基準ステータス（最大育成・限界突破）モデリング
- **内容**: キャラクターレベル、スタイルレベル、アビリティツリーがすべて「最大」まで上昇していることを前提とし、限界突破段階（`limitBreakCount`: 0〜4凸）のみを入力値とするステータス算出式を設計します。
- **インプット**: `characters.json` / `styles.json` 内の最大レベル時ステータスデータおよび `limit_break`（限界突破ボーナス）データ。
- **アウトプット**: ステータス算出関数 `calculate_max_base_stats(chara_id, style_id, limit_break_count)`。
- **完了条件**: 指定された限界突破段階における、最大育成・無装備状態のステータス値と完全に一致すること。

#### T1.3: 最上位テンプレート装備の補正補完ロジックの構築
- **内容**: 簡易計算時に適用する「最大レベル帯（Lv120以上）における標準的な最上位装備セット」を仮定し、ステータス上昇値を自動加算するロジックを作ります。
  - *仮定する最上位装備テンプレート例*:
    - 最高ランクブースター、属性リング（各属性+10%）、能力ブレスレット、および宝珠ボード全解放（全能力値+10%相当の補正）。
- **アウトプット**: 装備補正補完関数 `apply_max_gear_template(base_stats)`。
- **完了条件**: 一般的なプレイヤーが最大レベル帯で所持している標準的な実戦ステータスと、擬似算出されたステータスの誤差が極小であること。

#### T1.4: 敵防御境界値（param_border）の自動引き当て
- **内容**: シミュレータ側で敵が選択された際に、`enemies.json` から敵のレベルやIDに対応する `param_border` を自動参照するロジックを作ります。
- **アウトプット**: 敵防御情報取得関数 `get_enemy_border(enemy_id)`。
- **完了条件**: 指定した敵の防御境界値が、Excelの仮想敵シートのVLOOKUP相当値と完全一致すること。

### 📂 フェーズ2: データ接続仕様の確定とバフ・デバフ効果解決器の設計

シミュレータが管理するバフ・デバフの「付与状態」から、計算機が必要とする「最終倍率値」へ集約するロジックを設計します。

#### T2.1: バフ・デバフ効果量（知性・運依存）の動的解決器の設計
- **内容**: バフ/デバフのスキルIDから `skills.json` を引き当て、基礎効果量に対して使用者（バッファー/デバッファー）の知性・運に応じたスケーリング計算を行う関数を設計します。
- **アウトプット**: `resolve_buff_power(skill_id, provider_stats)` / `resolve_debuff_power(skill_id, provider_stats)`。
- **完了条件**: 各種バフスキル（専用・通常）が、使用者の知性に応じて正しい効果倍率に変換されること。

#### T2.2: 同一カテゴリバフ・デバフの重複上限（上位2枠）適用ロジック
- **内容**: ゲーム仕様に基づき、同じ「攻撃力バフ」「防御デバフ」が複数かかっている場合、効果量の高い上位2枠のみを適用して集約するロジックを設計・実装します。
- **アウトプット**: バフデバフ集約フィルター関数。
- **完了条件**: 3つ以上のバフが重複した際、下位のバフが正しく除外されて期待通りの集約値（`AJ67` / `AJ82` に相当）が得られること。

#### T2.3: `DamageInputContext` および `DamageResult` のJSONスキーマ確定
- **内容**: 接続フックで使用する最終的な入出力JSON仕様を確定し、ドキュメントとして合意します。
- **成果物**: インターフェース定義書（TypeScript型定義の確定）。

### 📂 フェーズ3: 計算エンジンのJS/TS移植とクロス言語検証テスト

PythonでのロジックをJS/TSに移植し、計算誤差がゼロであることを自動検証します。

#### T3.1: PythonモジュールからJS/TSモジュールへの移植
- **内容**: フェーズ1およびフェーズ2で構築したPythonコードを、シミュレータに内蔵可能な TypeScript (`hbr-damage-engine.ts`) に移植します。
- **アウトプット**: `hbr-damage-engine.ts`。
- **完了条件**: コンパイルエラーがなく、型定義に準拠していること。

#### T3.2: テストデータ発生器（Python）の作成
- **内容**: Pythonのテスト環境で、ランダムに様々なスタイル、レベル、バフ付与状況、敵を設定した「テスト入力パラメータ」と、Excelが計算した「期待される通常/クリティカルダメージ、およびブレイクダウン内訳値」をセットにしたテスト用データ（`test_cases.json`、1万ケース以上）を自動出力するスクリプトを作成します。
- **成果物**: `generate_test_cases.py` および出力された `test_cases.json`。

#### T3.3: クロス言語アサーションテストの実行
- **内容**: 移植した `hbr-damage-engine.ts` (JS/TS) に `test_cases.json` を読み込ませ、すべてのテストケースでJS側の出力がPython版（およびExcel期待値）と1円もズレずに完全一致するかを検証するテストランナーを実行します。
- **完了条件**: 100%のテストケースが成功（アサーションエラーなし）すること。

### 📂 フェーズ4: シミュレータへの組み込みとUI結合

完成した計算エンジンをシミュレータに接続し、画面上で結果を表示できるようにします。

#### T4.1: シミュレータのターン進行処理への接続
- **内容**: `hbr_battle_simulator` の `src/turn/turn-controller.js` でアクション実行時、または履歴再生時に、`DamageCalculationHook` の具現として移植したエンジンを呼び出す接続コードを実装します。
- **完了条件**: キャラクター行動時に、シミュレータ内部で正しくダメージコンテキストが計算されること。

#### T4.2: 威力増加ブレイクダウンパネルUIとのデータ連携
- **内容**: 計算結果に含まれる `breakdown`（内訳データ）を受け取り、[damage_breakdown_design.md](file:///Users/ram4/git/hbr_battle_simulator/docs/specs/damage_breakdown_design.md) に基づく詳細表示UIに値をバインドします。
- **完了条件**: キャラクターを右クリックした際などの詳細ポップアップに、攻撃バフ枠やクリティカル心眼枠などの実倍率と内訳が正しく描画されること。

#### T4.3: シミュレータ統合テスト（Playwright / 手動）
- **内容**: 実際のブラウザ上で、キャラクターを選択し、スキルを使い、バフをかけた状態で、期待通りのダメージが算出・表示され、ターンが正しく確定できるかをPlaywrightおよび手動で検証します。
- **完了条件**: すべての統合テストがエラーなしでパスすること。
