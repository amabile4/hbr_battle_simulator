# HBR計算機 ダメージ計算仕様・データ構造レポート

本ドキュメントは、「ダメージ計算機」シートのUIレイアウト、インプット/アウトプット構造、人間が入力しやすくするための工夫（UX設計）、および最終的なダメージ計算数式モデルを整理した仕様書です。

---

## 1. シートの役割とUX設計の特徴

「ダメージ計算機」シートは、複雑なゲーム内ダメージ計算式（攻撃力・知性・精神などのパラメータ、バフ・デバフ上限、クリティカル、敵耐性・破壊率等）を、ゲームプレイヤーが直感的に入力してシミュレーションできるように設計されています。

### ① UIレイアウトの明確なエリア分離
* **ユーザー入力・表示エリア（AF列〜BB列付近）**:
  * ユーザーが操作するバフ・デバフの選択、攻撃者、敵の設定がまとまって配置されています。
* **内部計算・定数エリア（A列〜AE列付近）**:
  * 画面左側の見えない領域に、データベース（マスタ）からの値の引き当て、中間式の評価、アビリティや宝珠の効果判定などの重いロジックが隠されています。

### ② 入力の手間を極限まで減らす工夫
* **データバリデーション（プルダウン）の連携**:
  * キャラクター（主力）を選択すると、そのキャラクターが持つスキルだけがスキル選択欄（`AZ20`）にフィルタリングされて表示されます。
  * 「編成セット」シートで登録したパーティを `AF17` で選択するだけで、バフやデバフの適用候補が自動的に絞り込まれます。
* **「能力値」シートとの連携**:
  * 攻撃者のステータス（力、器用さ等）やバフ・デバフ使用者のステータス（知性、運）は、「能力値」シートにあらかじめ入力されたデータから自動参照されます。手動で毎回数値を打ち直す必要がありません。
* **ゲーム内説明テキストからの自動数値抽出**:
  * バフ・デバフやスキルの特効（例: 「対HP+50%」など）は、テキストから `REGEXEXTRACT` を用いて自動的に数値（`50%`）を取り出して計算式に組み込んでいます。マスタデータのカラム数を増やさず、ゲーム内テキストをコピペするだけで動作するようになっています。

| :--- | :--- | :--- | :--- |

| `AF17` | 編成セット名 | リスト選択 | 現在のパーティ編成。「編成セット」シートを参照 |
| `AZ18` | 攻撃キャラクター | リスト選択 | 攻撃を担当するキャラクター |

### 中間式 (計算プロセス)
| セル座標 | 項目名 | 計算式 / 説明 |
| :--- | :--- | :--- |
| `AJ8` | 適用能力値 | スキルごとの参照ステータス比率（力・器用さ等の割合）を判定して算出された攻撃者ステータス。 |
| `W59` | 基礎ダメージ（通常） | 適用能力値と敵の防御力（精神）との差から算出される、バフ・デバフ適用前の基礎ダメージ。 |
| `X59` | 基礎ダメージ（クリ） | クリティカル時は敵防御力が減少した（-50デバフ相当）状態として計算される基礎ダメージ。 |
| `AJ88` | 耐性/弱点累計 | 属性耐性、弱点補正、および「貫通クリティカル」等の乗算補正の合計。 |

| `AJ67` | バフ累計 | 各攻撃バフの効果量の合計。 |
| `AS42` | トークン累計 | 所持トークン数に応じたダメージ上昇率。 |
| `AR65` | クリダメバフ累計 | クリティカル時のダメージアップバフ（コンセントレーション等）の合計。 |

| セル座標 | 項目名 | 数式 | 説明 |
| :--- | :--- | :--- | :--- |
| **`Y59`** | **通常ダメージ期待値** | `=W59 * AJ88 * AJ10 * AJ11 * (AJ82+100%) * (Vulnerability+100%) * (AJ67+100%) * (AS42+100%) * Funnel` | バフ・デバフ・特効・脆弱・連撃等をすべて乗算した通常ダメージの平均値。 |

## 3. ダメージ計算の数式モデル（数式化）

シート上の複雑なセル参照を、プログラム移植しやすいように一般的な変数式にモデル化します。

### ① 通常ダメージ期待値の計算モデル

$$Damage_{normal} = BaseDamage_{normal} \times Resistance \times BreakRate \times SpecialEffect \times (1 + Debuff) \times (1 + Vulnerability) \times (1 + Buff) \times (1 + Token) \times Funnel$$

* **$BaseDamage_{normal}$ (基礎通常ダメージ / `W59`)**:
  * 攻撃側ステータス $Status_{atk}$ と敵防御ステータス $Status_{def}$ の差に基づきます。
  * $Status_{atk} \ge Status_{def}$ の場合と下回る場合で計算式（閾値）が切り替わります。
* **$Resistance$ (弱点・耐性乗算補正 / `AJ88`)**:
  * スキル属性と敵耐性による乗算。
* **$BreakRate$ (破壊率補正 / `AJ10`)**:
  * 敵の現在の破壊率がそのまま乗算されます（例: 破壊率250% = 2.5倍）。
* **$SpecialEffect$ (対HP/DP特効 / `AJ11`)**:
  * 攻撃対象がHPかDPかによって、該当する特効値（+50%など）が加算されます。
* **$Debuff$ (防御デバフ累計 / `AJ82`)**:
  * 適用されている防御デバフ率の合計値。
* **$Vulnerability$ (脆弱デバフ累計)**:
  * 防御デバフとは別乗算枠となる脆弱デバフ（`Fragile`）の合計値。通常脆弱（弱点時のみ有効）と永続脆弱の合計です。
* **$Buff$ (攻撃・属性バフ累計 / `AJ67`)**:
  * 適用されている攻撃バフ・属性バフ率の合計値。
* **$Token$ (トークン補正 / `AS42`)**:
  * トークンによるダメージ上昇率。
* **$Funnel$ (連撃補正)**:
  * 連撃バフ（`Funnel`）の効果によるダメージ上昇率。

### ② クリティカルダメージ期待値の計算モデル

クリティカル時は、**敵の防御ステータス（防御境界値）を減少**させた状態で基礎ダメージが計算され、さらに**クリティカルダメージアップバフおよび心眼バフ**が乗算されます。

$$Damage_{crit} = BaseDamage_{crit} \times Resistance \times BreakRate \times SpecialEffect \times (1 + Debuff) \times (1 + Vulnerability) \times (1 + Buff) \times (1 + Token) \times \frac{1.5 + CritBuff + MindEye}{1.5} \times Funnel$$

#### クリティカル時防御境界値の算出式 ($border\_crit$)

クリティカル時の敵防御境界値 $border\_crit$ は、アビリティ等による精神補正 $abilitySprCorrection$ を考慮して以下のように算出されます。

$$border\_crit = param\_border - 50 - \max(0, -50 - abilitySprCorrection)$$

- $param\_border$: 敵の防御境界値（防御力・精神）。
- $abilitySprCorrection$: アビリティ等による精神補正（デフォルト: `0.0`、マイナス値も許容）。
- **補正値のクランプ挙動**: アビリティ補正が `-50` を下回る（さらに強力に減少させる）場合、追加でその差分が減少量に加算されます（例: `-80.0` が指定された場合、減少値は `-50 - (max(0, 30)) = -80` となり、防御力は 80 減少します）。

* **$BaseDamage_{crit}$ (基礎クリティカルダメージ / `X59`)**:
  * 敵の防御ステータスを上記の $border\_crit$ に引き下げた状態で計算される基礎ダメージ。通常攻撃のみクリティカル下限補間に $E/2$ を使用します。
* **$CritBuff$ (クリティカルダメージバフ累計 / `AR65`)**:
  * クリティカル時の基礎ダメージ倍率 `1.5` (150%) に対し、追加のクリティカル威力バフ（例: +20%）を加算した倍率補正。
* **$MindEye$ (心眼バフ累計)**:
  * クリティカル時の基礎倍率に加算される心眼効果（弱点攻撃時のみ有効）。$CritBuff$ と同様に基礎倍率 `1.5` に対する加算要素となります（非弱点攻撃時は無効となり `0` として処理されます）。

## 4. 計算倍率 (Multiplier) 詳細仕様定義

最終ダメージ計算式で使用される各種乗算値 (Multiplier) の定義と入力マッピングは以下の通りです。

| 倍率名 | 入力ソース (型定義) | 計算数式 | デフォルト値 | 制限・特記事項 |
| :--- | :--- | :--- | :--- | :--- |
| **$Resistance$**<br>(耐性乗算補正) | `defender.resistances`<br>`activeZone` | `affinityMultiplier * zoneMultiplier` | `1.0` | 武器属性相性とゾーン倍率の乗算（ゾーン一致時は 1.5 倍）。1.0 を超えた場合に弱点攻撃と判定。 |
| **$BreakRate$**<br>(破壊率補正) | `defender.destructionRate` | `destructionRate` (そのまま乗算) | `1.0` | 入力は倍率表記（例: `2.5` = 破壊率 250% を表す）。下限: `1.0`。 |
| **$SpecialEffect$**<br>(対HP/DP特効) | スキルの威力定義<br>(`defender.isHpTarget` 依存) | `parts[].multipliers.hp` (または `dp`) | `1.0` | 対象がHPゲージなら `hp` 特効値、DPゲージなら `dp` 特効値（例: HP+50%時は `1.5`）を適用。 |
| **$Debuff$**<br>(防御デバフ累計) | `defender.statusEffects`<br>`defender.passiveDefenseDown` | `aggregateDebuffs(debuffs) + passiveDefenseDown` | `0.0` (補正 1.0) | カテゴリ別の重複上限を適用。`passiveDefenseDown` はカテゴリ上限（上位2枠）から独立して全加算。 |
| **$Vulnerability$**<br>(脆弱デバフ累計) | `defender.statusEffects` | `aggregateFragiles(fragiles, isWeakness)` | `0.0` (補正 1.0) | 通常脆弱は弱点攻撃時（$Resistance > 1.0$）のみ有効。永続脆弱は常時有効。それぞれ上位2枠まで合計。 |
| **$Buff$**<br>(攻撃バフ累計) | `attacker.statusEffects` | `aggregateBuffs(buffs)` | `0.0` (補正 1.0) | 通常バフ（上位2枠合計）と単独発動バフ（最大効果1枠）の大きい方を適用。 |
| **$Token$**<br>(トークン補正) | `attacker.tokenRatio`<br>`attacker.tokenCount` | `tokenRatio` (指定なし時は `tokenCount * 0.10`) | `0.0` (補正 1.0) | トークンによるダメージ上昇率。 |
| **$Funnel$**<br>(連撃補正) | `attacker.statusEffects` | `1.0 + sum(funnel_resolved_power) / 100` | `1.0` | 連撃バフの効果量（%）の単純合計値を乗算。上限なし。 |

## 5. 特殊仕様および将来の課題 (WIP)

### ① パラメータ限界値超過時の挙動オプション
通常およびクリティカルの基礎ダメージ計算において、キャラクターの適用能力値が限界値（`Enemy Border + limit`）を超過した際の処理をオプションで切り替え可能です。
* **クランプモデル（`options.clampOverLimit: true`、デフォルト）**:
  * スプレッドシート（Excel）と完全に同一の仕様。限界値に達した時点でダメージは完全に頭打ち（`maxPower`）になります。
* **超過成長モデル（`options.clampOverLimit: false`）**:
  * `hbr-tool.com` および実機に準拠した仕様。限界値を超過した際、1ポイントの超過につき `maxPower * 0.0025` （0.25%）の割合でダメージが緩やかに上昇し続けます。

### ② ヒット単位での破壊率蓄積とダメージ連動シミュレーション (WIP)
* **現状の仕様**: スキル攻撃時のダメージ計算において、攻撃開始時点の破壊率（`BreakRate` / `AJ10`）を全ヒットに対して一括適用して総ダメージを算出しています。
* **実際のゲーム仕様 / 将来の課題**: 
  * ダメージはスキルのヒット数（`hit_count`）およびその配分比率（`damege_distribution`）に従ってヒットごとに分割処理されます。
  * HPゲージを攻撃中、各ヒットが与えた破壊率の上昇分（`Destruction Rate`）は、そのスキル内の**次のヒットの計算に即座に反映**され、ダメージを押し上げる要因となります。
  * この「ヒット単位での破壊率の動的更新とダメージ計算の連動」は、破壊率の基本エンジンの動作検証が完了した後の追加課題（WIP）として、将来対応予定とします。
