# HBR計算機スプレッドシート 構造解析・検証レポート

本ドキュメントは、非常に複雑なGoogleスプレッドシート（以下、「HBR計算機」）の計算式を分析・抽出し、他ツール（Python等）でライブラリ化して転用するための、構造解析結果と数式復元の妥当性検証レポートです。

---

## 1. 概要
* **解析対象ファイル**: 
  * `HBR計算機🎭Ver.4.31.03.xlsx` (Excelエクスポート版)
  * `HBR計算機🎭Ver.4.31.03.ods` (OpenDocumentエクスポート版)
* **目的**: 
  * 計算ロジックを丸裸にし、シート構造・数式の依存関係・使用関数を明確にする。
  * エクスポート時に生じる数式の破損（UDF化）を正しく復元できているかを別形式（ODS）とクロス検証する。

## 3. 計算式復元の検証と妥当性（XLSX vs ODS）

### Google Sheetsからエクスポートした際の課題
GoogleスプレッドシートからExcel（.xlsx）へエクスポートする際、Google Sheets固有の関数（`REGEXMATCH` や `ARRAYFORMULA` 等）や極めて複雑なネストがある数式は、Excelでサポートされないため以下のようなUDF（User Defined Function）表記に自動変換され、そのままでは直接パースできません。

```excel
# XLSX内での元の格納状態
=IFERROR(__xludf.DUMMYFUNCTION("IMPORTRANGE(""1VIA..."",""シート名!A:H"")"), "")
```

### 復元処理（クリーンアップ）
この `__xludf.DUMMYFUNCTION("...")` 内にエスケープされて閉じ込められている本来のGoogleスプレッドシートの式を抽出し、アンエスケープして元の数式を復元しました。

### ODS（OpenDocument）によるクロス検証
復元ロジックに誤り（バグ）がないかを検証するため、別経路でエクスポートした `.ods` ファイルの `content.xml` 内に定義されている数式と、XLSXから復元した数式の全セル比較を行いました。

#### 比較時の差分と論理的一致の証明
ODSの生の数式とXLSX復元数式の間には、シリアライズ仕様に起因する表記の差分が存在しますが、**論理的な構造は完全に同一**であることが実証されました。

* **差分1：引数の区切り文字**
  * XLSX: カンマ `,` (例: `VLOOKUP(A1, B1:C10, 2, FALSE)`)
  * ODS: セミコロン `;` (例: `VLOOKUP(A1; B1:C10; 2; 0)`)
* **差分2：セル・シートの参照文法**
  * XLSX: `'シート名'!A:G`
  * ODS: `[$シート名.A$1:.G$1048576]`
* **差分3：論理値と数値の揺れ**
  * ODSでは、`TRUE` が `1`（IFSのデフォルト条件など）、`FALSE` が `0` として出力されるケースがあります。

#### 比較例（スコアタ計算機 P13セル）
* **XLSXから復元した数式:**
  ```excel
  =(ROUNDDOWN(IF(P8<VLOOKUP(P4,C4:M105,11,FALSE),(VLOOKUP(P3,A67:B69,2,FALSE))*P8,(VLOOKUP(P3,A67:B69,2,FALSE))*(VLOOKUP(P4,C4:M105,11,FALSE)*LN(P8/VLOOKUP(P4,C4:M105,11,FALSE))+VLOOKUP(P4,C4:M105,11,FALSE))),0))
  ```
* **ODSから抽出した数式:**
  ```excel
  # Raw XML
  of:=(ROUNDDOWN(IF([.P8]<VLOOKUP([.P4];[.C4:.M105];11;0);(VLOOKUP([.P3];[.A67:.B69];2;0))*[.P8];(VLOOKUP([.P3];[.A67:.B69];2;0))*(VLOOKUP([.P4];[.C4:.M105];11;0)*LN([.P8]/VLOOKUP([.P4];[.C4:.M105];11;0))+VLOOKUP([.P4];[.C4:.M105];11;0)));0))
  ```
  表記（`,` と `;` や `[]`）を除き、ロジックツリーは完全に一致しています。

## 4. ライブラリ化（他システムへの移植）における技術的課題と対策

本スプレッドシートの数式をPythonなどの別システムへ移植・ライブラリ化するにあたり、以下の技術的課題に対処する必要があります。

### ① Googleスプレッドシート固有関数への対応
* **`REGEXMATCH` / `REGEXEXTRACT`**
  * **課題**: スタイル判定やバフ・デバフ効果のテキスト解析に多用されています。Excelには存在しません。
  * **対策**: Python移植時は `re` モジュールの `re.search` または `re.match` を使用した処理にマッピングします。
* **`ARRAYFORMULA`**
  * **課題**: セル範囲全体に一度に数式を適用する機能です。
  * **対策**: 単純なループ処理にするか、Pandasのベクトル演算（Series/DataFrameに対する演算）に置き換えます。
* **`IMPORTRANGE`**
  * **課題**: 他のスプレッドシートから動的にデータをロードしています。
  * **対策**: 参照先スプレッドシート（マスタデータなど）のデータを事前にJSON/CSV/DB等に保存し、ライブラリ内部で結合（Join）処理を行います。

### ② 依存関係の解決（計算順序の決定）
* **課題**: シート内のセルが相互に参照し合っているため、正しい順番で計算を評価しなければ結果がズレます。
* **対策**: `networkx` などのグラフライブラリを用いて、セル間の依存関係を **有向グラフ (DAG)** としてモデル化し、**トポロジカルソート**を行うことで、正しい計算順序を自動的に決定します。

### ③ 自動検証テスト（リグレッションテスト）の自動生成
* **課題**: 2万個近い数式ロジックをすべて手動でテストするのは不可能です。
* **対策**:
  1. `openpyxl` を用いて、インプット値（ステータスやバフ等）を様々に書き換えたテストケースファイルを自動生成。
  2. XLSXの計算エンジンで得られる期待値（`data_only=True` でロードしたセルの値）を保存。
  3. 移植後のPythonライブラリに同じパラメータを入力し、出力値が期待値と完全に一致（許容誤差範囲内）するかを検証するテストスイートを実行。
