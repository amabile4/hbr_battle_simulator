## 1. ターン制御システムの実現可能性：【高】
- **状態機械の堅牢性**: `normal` / `od` / `extra` の状態遷移は、DEC-004の `ExtraTurnState` 構造化により、再帰的な実装を避けつつフラットなステート管理が可能になりました。
- **Preview/Commit分離**: 副作用のない `previewTurn` と、状態を確定させる `commitTurn` の分離は、現在の `js/control-manager.js` の構造を整理し、UIでの「行動予約」と「実行」を明確に区別できるため、実装難易度は低いと判断します。
- **同時フラグ管理**: `odPending`（OD予約）と `odSuspended`（割り込み不可）の同時保持は、ターン終了処理のパイプライン（後述）の最後に評価することで、決定論的な挙動を保証できます。

## 2. OD/extra 複雑シナリオの処理可能性：【高】
- **連続Extraの制御**: `allowedCharacterIds` を `ExtraTurnState` 内で管理し、追加発生時に集合（Set）としてマージする設計により、複雑な行動順の上書きもロジックの破綻なく処理可能です。
- **OD発動の遅延実行**: 「全Extra完了後にOD発動」というルールは、`TurnController` の終了判定ルーチンに `if (!extraState && odPending) triggerOD()` を組み込むことで簡潔に実装できます。
- **Context管理**: `odContext` を `preemptive` / `interrupt` で明示化したことにより、OD終了時のターン復帰位置の特定が容易になります。

## 3. SP変動パイプライン：【中（注意深い実装が必要）】
- **処理順序の厳密性**: Cost → Base → OD → Passive → Clamp の順序実行は、関数合成（Pipeline）パターンで実装可能です。
- **R10凍結ルールの適用**: `effectiveCeiling = Math.max(current, eventCeiling)` のロジックは、SPが上限を超えている場合（バフ等）の「減少はさせないが、上限以下の回復は無視する」という挙動を正確に記述しており、計算式のミスを防げます。
- **OD中の上限分離**: `source='od'` の時のみ `ceiling=99` とし、それ以外は `sp.max` を参照する設計は、R10の「ODはsp.maxを変動させない」という原則をコードレベルで担保しています。
